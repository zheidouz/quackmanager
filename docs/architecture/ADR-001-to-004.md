# ADR-001: Use React + Vite over Next.js

**Status**: Accepted
**Date**: 2026-06-02
**Deciders**: Project Owner

## Context
QuackManager is a pure offline-first PWA with no need for server-side rendering, API routes, or SEO. Next.js adds unnecessary complexity and potential cost (server hosting) for this use case.

## Decision
Use React 18+ with Vite as the build tool, and `vite-plugin-pwa` for service worker generation.

## Consequences
- **Positive**: No server costs, smaller bundle, faster builds, simpler deployment
- **Negative**: No SSR for potential future public pages (not a concern for internal tool)
- **Neutral**: Team may need to learn Vite-specific configuration

# ADR-002: Zustand + Dexie.js Dual-State Pattern

**Status**: Accepted
**Date**: 2026-06-02
**Deciders**: Project Owner

## Context
The app needs both transient UI state (active tab, sync status) and persistent data (egg counts, sales). Mixing them in a single state management solution causes complexity.

## Decision
Use Zustand for transient UI state only. All persistent farm data goes through Dexie.js (IndexedDB wrapper).

## Consequences
- **Positive**: Clear separation of concerns; Dexie provides offline-first reads instantly
- **Positive**: Zustand stores stay small and focused (authStore, appStore, alertStore, syncStore)
- **Negative**: Developers must understand both systems and the boundary between them

# ADR-003: Firebase Serverless over Custom Backend

**Status**: Accepted
**Date**: 2026-06-02
**Deciders**: Project Owner

## Context
A custom backend (Node.js + Express) would require server hosting, maintenance, and scaling. For a single-user internal tool, operational overhead must be minimized.

## Decision
Use Firebase (Firestore + Auth + Hosting) as a serverless backend. No custom server code.

## Consequences
- **Positive**: $0/month cost, no server management, built-in auth
- **Positive**: Firestore provides real-time sync capabilities
- **Negative**: Vendor lock-in to Firebase (acceptable for internal tool)
- **Negative**: Firestore query limitations compared to SQL

# ADR-004: Last-Write-Wins Conflict Resolution

**Status**: Proposed
**Date**: 2026-06-02
**Deciders**: Project Owner

## Context
Offline writes may conflict with cloud writes when connectivity is restored. A conflict resolution strategy is needed.

## Decision
Use last-write-wins based on `updatedAt` timestamp. This is the simplest strategy and is acceptable for a single-user app where conflicts are rare.

## Consequences
- **Positive**: Simple to implement, no conflict UI needed
- **Positive**: Predictable behavior
- **Negative**: If data entry happens on two devices simultaneously, the later write wins silently
